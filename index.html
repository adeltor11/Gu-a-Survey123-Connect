Plantilla XLSForm Avanzada

Guía completa para crear, configurar y publicar formularios de captura de datos en campo. Cubre cada columna, cada tipo de pregunta y la lógica de validación de la plantilla Advanced incluida en Survey123 Connect.

Versión Template 3.25 App Survey123 Field App + Web App Contexto EDECA · UNA · Costa Rica
01 · Introducción

¿Qué es esta plantilla y para qué sirve?

Esta plantilla es un archivo Excel (.xlsx / .xlsm) que actúa como el lenguaje con el que ArcGIS Survey123 Connect construye formularios digitales de captura de datos. El formato que usa se llama XLSForm, un estándar abierto desarrollado por Open Data Kit (ODK) y adoptado por Esri para Survey123.

Cada fila que se agrega en la hoja survey es una pregunta, un campo de datos o una instrucción de comportamiento. Survey123 Connect lee ese archivo y lo convierte en un formulario interactivo que se instala en dispositivos móviles (Android, iOS) o funciona desde el navegador.

ℹ
Versión de la plantilla Esta documentación corresponde a la versión Advanced 3.25. Es la versión completa que incluye todas las funciones soportadas, tanto en la Field App como en la Web App.

¿Por qué usar Survey123 en lugar de papel?

  • Captura de coordenadas GPS precisas directamente desde el dispositivo o receptor externo (Trimble TDC6, etc.).
  • Validación automática de datos en el momento de la medición: el formulario rechaza valores fuera de rango.
  • Cálculos automáticos en tiempo real: área basal, biomasa, factor de forma, directamente en campo.
  • Sincronización directa con ArcGIS Online, Feature Services y bases de datos PostgreSQL/PostGIS.
  • Eliminación de errores de transcripción al digitalizar datos de papel.
  • Adjuntar fotografías, audio y archivos directamente a cada registro georreferenciado.

Columna name

Es el nombre interno del campo, equivalente al nombre de columna en la tabla de datos del Feature Layer. Funciona como el nombre de una variable en programación.

⚠
Reglas obligatorias Solo letras minúsculas, números y guion bajo (_). Sin espacios, sin tildes, sin caracteres especiales. No puede comenzar con número. Debe ser único en todo el formulario. Máximo recomendado: 10 caracteres si se conecta a geodatabases heredadas.

Ejemplos para inventario forestal

  • parcela_id — Identificador de la parcela
  • dap_cm — Diámetro a la Altura del Pecho en centímetros
  • altura_m — Altura total del árbol en metros
  • especie — Selección de especie forestal
  • ubicacion — Campo geopoint
  • foto_arbol — Imagen del árbol

Columnas label, hint y guidance_hint

label

Texto visible que el usuario ve como título de la pregunta. Acepta cualquier carácter, incluyendo tildes, espacios, emojis y etiquetas HTML básicas.

→
Buena práctica Incluir la unidad de medida en el label. Si el name es dap_cm, el label debería ser: Diámetro a la Altura del Pecho — DAP (cm)

hint

Texto de ayuda que aparece debajo del label. Sirve para instrucciones de protocolo de medición. Ejemplo: "Medir a 1.30 m desde la base del árbol. Si hay contrafuertes, medir por encima de estos."

guidance_hint

Texto extendido que se despliega solo cuando el usuario toca un ícono de información adicional. Ideal para protocolos técnicos detallados que no deben ocupar espacio permanente en pantalla.

Columnas required, required_message y readonly

required

Si se coloca yes, el formulario no permite avanzar ni enviar sin que el campo tenga un valor.

  • yes o true — Campo obligatorio
  • no o false — Campo opcional (por defecto)
  • Expresión XPath — Obligatorio condicionalmente: ${dap_cm} > 0

required_message

Mensaje personalizado cuando el usuario intenta enviar sin completar un campo obligatorio. Si se deja vacío, Survey123 muestra un mensaje genérico en inglés.

readonly

Hace el campo visible pero no editable. Se usa principalmente para mostrar resultados de cálculos en pantalla. Valores: yes / no.

⚠
Importante En inventarios forestales, campos como especie, DAP y geopoint deben marcarse como required = yes para garantizar la calidad del dato capturado.

Columna default

Valor que se pre-carga automáticamente al abrir el formulario. El usuario puede modificarlo.

  • today() — Fecha actual (campos date)
  • now() — Fecha y hora actuales (campos dateTime / time)
  • Un valor fijo de texto o número
  • Referencia a otro campo: ${nombre_finca}
  • Para select_multiple: valores separados por coma: item1,item2

Columna calculation

La columna de fórmulas. Se usa junto con el tipo calculate para ejecutar operaciones matemáticas o lógicas. El resultado se guarda en la base de datos y puede mostrarse en pantalla si se combina con readonly = yes.

Operadores XPath

OperadorDescripciónEjemplo
+Suma${altura_m} + ${altura_fuste}
-Resta${altura_total} - ${altura_fuste}
*Multiplicación${dap_cm} * ${dap_cm}
divDivisión${peso_kg} div ${volumen_m3}
modMódulo (resto)${numero} mod 2
=Igual${especie} = 'teca'
!=Diferente${estado} != 'muerto'
< / <=Menor / Menor o igual${dap_cm} < 10

or / andDisyunción / Conjunción${dap_cm} > 5 and ${dap_cm} < 300

Funciones útiles

  • if(condición, valor_verdad, valor_falso) — Condicional simple
  • round(expresión, decimales) — Redondeo: round(${ab_m2}, 4)
  • concat(${campo1}, ' - ', ${campo2}) — Concatenar textos
  • string-length(${campo}) — Longitud de un texto
  • format-number(${valor}, '#.##') — Dar formato numérico
  • today() / now() — Fecha y hora actuales
  • pulldata('archivo', 'campo', 'clave', ${ref}) — Leer desde CSV externo

Ejemplo: área basal por árbol

3.14159265 * (${dap_cm} div 100 div 2) * (${dap_cm} div 100 div 2)

El resultado es el área basal en m². Para mostrarlo en pantalla: appearance vacío + readonly = yes.

Columnas constraint y constraint_message

Define una regla de validación. Si el valor ingresado no cumple la expresión, el formulario muestra un error y bloquea el envío. Es la herramienta principal de control de calidad de datos en campo.

ℹ
El punto (.) en una expresión constraint hace referencia al valor actual del campo que se está validando.

Ejemplos para inventario forestal

  • . >= 5 and . <= 300 — DAP válido entre 5 y 300 cm
  • . >= 1 and . <= 80 — Altura total entre 1 y 80 metros
  • . > 0 — El valor debe ser positivo
  • regex(., '^[A-Z]{2}-[0-9]{4}$') — Código con formato AB-1234
  • string-length(.) <= 100 — Texto de máximo 100 caracteres

constraint_message: mensaje personalizado cuando no se cumple la regla. Si se deja vacío, Survey123 muestra un texto genérico en inglés.

Columna relevant

Controla la visibilidad condicional. Si la expresión es verdadera, la pregunta se muestra. Si es falsa, se oculta y su valor no se envía. Permite crear formularios dinámicos e inteligentes.

Ejemplos

  • ${dap_cm} > 0 — Muestra campos de biomasa solo si se midió el DAP
  • ${estado_arbol} = 'vivo' — Campos fitosanitarios solo para árboles vivos
  • ${tiene_dano} = 'yes' — Sub-campos de daño solo si se marcó que hay daño
  • ${tipo_medicion} = 'parcela' — Campos de parcela solo en ese modo
→
El uso inteligente de relevant hace el formulario más corto y amigable para el dasónomo en campo, mostrando solo lo que es pertinente según las respuestas anteriores.

Columna choice_filter

Filtra la lista de opciones de un select_one o select_multiple basándose en atributos adicionales definidos en la hoja choices. Habilita los selectores en cascada.

Ejemplo forestal

  • Primera pregunta: selección de Género (Tectona, Pinus, Cedrela...)
  • Segunda pregunta: selección de Especie, filtrada por el género elegido
  • La expresión en choice_filter: genero = ${genero_elegido}
  • En la hoja choices se agrega una columna genero con el valor correspondiente para cada especie

Columna repeat_count

Define el número de repeticiones de un bloque begin repeat. Si se deja vacío, el usuario agrega repeticiones manualmente. Si se coloca un número o expresión, el formulario crea ese número de bloques automáticamente.

  • Vacío — El usuario agrega y elimina repeticiones con botones
  • 10 — Exactamente 10 repeticiones fijas
  • ${num_arboles} — El número depende de una respuesta anterior

Columnas de idioma adicional

Las columnas con el sufijo ::language (xx) permiten agregar traducciones del texto a un idioma adicional: label::English (en), hint::English (en), etc. El idioma adicional se configura en la hoja settings.

ℹ
Si el formulario es solo en español, estas columnas se dejan vacías. No es necesario eliminarlas.

06 · Lógica avanzada

Grupos y repeticiones

begin group / end group

Todo lo que esté entre begin group y end group pertenece a ese grupo. Se usan para:

  • Organizar el formulario en secciones lógicas temáticas
  • Aplicar relevant a todo el grupo (mostrar u ocultar secciones completas)
  • Con style = pages, cada grupo con appearance = field-list se convierte en una página
  • Usar appearance = table-list para preguntas relacionadas en formato tabla

begin repeat / end repeat

Fundamental para inventarios forestales. Permite registrar múltiples árboles por formulario sin necesidad de abrirlo una vez por árbol. El colector completa la información de parcela una sola vez y repite el bloque de árbol las veces necesarias.

⚠
Dato importante Cada repetición se almacena en una tabla relacionada (relación 1:N) respecto al registro principal. Esto requiere diseño previo del esquema del Feature Service en ArcGIS Online.

XPath y la función pulldata()

XPath es el lenguaje de expresiones que Survey123 usa en calculation, constraint, relevant, required y default.

Principios básicos

  • Referenciar otro campo: ${nombre_del_campo}
  • Referenciar el valor propio (en constraint): . (punto solo)
  • Texto en comparaciones entre comillas simples: ${especie} = 'teca'
  • Números sin comillas: ${dap_cm} > 10

pulldata()

Lee datos desde un CSV externo en la carpeta media. Útil para precargar información de inventarios previos, catálogos de especies o datos de parcelas.

pulldata('nombre_archivo', 'columna_a_retornar', 'columna_de_busqueda', ${campo_referencia})

Ejemplo

CSV parcelas.csv con columnas id_parcela, area_ha, propietario:

pulldata('parcelas', 'area_ha', 'id_parcela', ${parcela_id})
⚠
Incompatibilidad conocida pulldata() puede tener conflictos con format-number() en Field App v3.18+. Usar round() en su lugar para formatear el resultado numérico de un pulldata().

07 · Referencia

Flujo de trabajo paso a paso

01

Planificar el formulario

Definir en papel qué datos capturar: secciones, preguntas, tipos de dato, campos obligatorios, validaciones y listas de opciones necesarias.

02

Configurar la hoja settings

Completar: form_title, form_id, instance_name, version y style.

03

Crear las listas en choices

Definir todas las listas de opciones. Una lista por cada pregunta de selección. Verificar que los name no tengan espacios ni caracteres especiales.

04

Construir el formulario en survey

Agregar campos de control (start, end, username), luego los grupos y dentro de ellos las preguntas.

05

Agregar validaciones y lógica

Revisar campos obligatorios (required), rangos (constraint), visibilidad condicional (relevant) y cálculos automáticos (calculate).

06

Guardar el archivo

Guardar en formato .xlsx. Survey123 Connect acepta ambos (.xlsx y .xlsm), pero .xlsx es más compatible.

07

Importar en Survey123 Connect

New Survey → File. Survey123 lee la plantilla y muestra una vista previa del formulario. Revisar el panel de errores.

08

Revisar y corregir errores

Los errores más comunes: names duplicados, apariencias incompatibles con el tipo de pregunta, referencias a listas inexistentes, expresiones XPath mal escritas.

09

Publicar

Clic en Publish. Survey123 crea automáticamente un Feature Layer en ArcGIS Online con el esquema definido. El formulario queda disponible para descargarse en dispositivos de campo.

10

Actualizar versiones

Editar el .xlsx, incrementar version en settings, y volver a importar y publicar. Survey123 detecta la nueva versión y actualiza en todos los dispositivos.

Appearance inválidaLa apariencia no es compatible con el tipo de pregunta. Consultar la hoja Appearances del template.

format-number() con pulldata() fallaIncompatibilidad conocida en Field App. Usar round() en lugar de format-number().
GPS no capturaLa aplicación no tiene permisos de ubicación en el dispositivo. Verificar permisos del sistema.
accuracyThreshold bloquea el GPSEl umbral es muy estricto para el entorno (dosel cerrado). Ajustar a 1.0 m o 2.0 m según condiciones.
Repeat no guarda todas las filasEl begin repeat no tiene su end repeat correspondiente, o hay inconsistencia en los niveles de agrupación.
calculate no actualiza su valorDepende de campos que aún no tienen respuesta. Agregar relevant para que solo calcule cuando los campos dependientes estén llenos.

Buenas prácticas

Nomenclatura de campos

  • Nombres descriptivos y cortos, sin tildes.
  • Incluir la unidad en el nombre: dap_cm, altura_m, peso_kg.
  • Evitar nombres genéricos como campo1 o dato2.

Calidad de datos

  • Aplicar constraint con rangos realistas para todos los campos numéricos dasométricos.
  • Usar required = yes en todos los campos que forman la llave primaria del registro.
  • Usar bind::esri:warning (no constraint) para valores inusuales pero posibles.
  • Incluir siempre un campo de observaciones de texto libre para situaciones no previstas.

Usabilidad en campo

  • Usar minimal para listas largas y autocomplete para catálogos de más de 20 opciones.
  • Organizar en páginas (style = pages) para evitar scroll excesivo.
  • Agregar notas (type = note) con instrucciones de protocolo entre secciones.
  • Configurar default = today() en el campo de fecha para agilizar la captura.

Control de versiones

  • Incrementar version en settings antes de cada publicación.
  • Guardar el .xlsx con la fecha en el nombre: inventario_arboreo_2025-01-15.xlsx.

Compatibilidad Field App vs Web App

Las siguientes funciones no están disponibles en la Web App:

  • week-number (date)
  • distress, spinner, numbers, calculator para integer/decimal
  • rangefinder (requiere hardware adicional)
  • deviceid

Glosario

XLSForm
Estándar abierto para definir formularios de recopilación de datos en hojas de cálculo Excel. Es el formato base de esta plantilla.
Survey123 Connect
Aplicación de escritorio de Esri para diseñar, probar y publicar formularios XLSForm en la plataforma ArcGIS.
Feature Layer
Capa de datos geográficos en ArcGIS Online donde se almacenan los registros enviados desde el formulario.
XPath
Lenguaje de consulta y expresión usado en Survey123 para referenciar campos, hacer cálculos y definir condiciones.
Cascading select
Técnica donde una lista de opciones se filtra según la respuesta de otra pregunta anterior. Requiere choice_filter en survey y atributos adicionales en choices.
Repeat
Bloque de preguntas que puede repetirse N veces. Los datos se almacenan en una tabla relacionada (relación 1:N) en el Feature Service.
esriFieldType
Tipos de campo de la geodatabase de Esri que definen cómo se almacenan los datos en el Feature Layer.
pulldata()
Función XPath que extrae un valor de un archivo CSV externo basándose en una clave de búsqueda.
geopoint
Tipo de pregunta que captura coordenadas geográficas (latitud, longitud, altitud) desde el GPS del dispositivo o receptor externo.
bind::
Prefijo de columnas que conectan la pregunta con el esquema de almacenamiento del Feature Layer.
body::
Prefijo de columnas que controlan la presentación visual del elemento en la pantalla del formulario.
constraint
Expresión XPath que valida el valor ingresado. Si no se cumple, bloquea el envío del registro.
relevant
Expresión XPath que controla si una pregunta se muestra o se oculta dinámicamente según respuestas anteriores.
accuracyThreshold
Umbral de precisión GPS (en metros) que el dispositivo debe alcanzar para permitir la captura de coordenadas.
media folder
Carpeta del formulario donde se almacenan archivos de referencia (imágenes, audio, CSV) utilizados en la plantilla.
instance_name
Expresión XPath que genera el nombre de cada registro enviado. Facilita identificar los envíos en ArcGIS Online.
pages (style)
Modo de navegación del formulario donde cada grupo con field-list se muestra en una página separada.
